# Бархат — навигация

> Читай этот файл в начале каждой сессии.

## Что это за проект

**Бархат** — сеть цветочных салонов. Занимаемся изготовлением и продажей букетов из цветов, клубники в шоколаде и других подарков.
Текущий статус: бизнес запущен и работает, в стадии масштабирования. Цель: автоматизировать процессы, увеличить выручку и прибыль, улучшить маркетинг.

## Ключевые принципы

1. **Минимум ручного труда** — всё, что можно автоматизировать, должно быть автоматизировано.
2. **Оптимизированные процессы с максимальным качеством** — эффективность не должна идти в ущерб качеству продукта и сервиса.

## КРИТИЧЕСКИ ВАЖНО: принцип автоматизации

**Владелец работает один. Ручной труд = враг роста.**

- Если задачу можно автоматизировать — она автоматизируется. Без исключений.
- Человек принимает только решения. Всё остальное — агенты и AI.

**При проектировании любой задачи спрашивай:**
> «Что здесь делает человек, что мог бы сделать агент?»

## Как работать с планами (цикл 3 чатов)

```
Чат 1: создать черновой план + поиск аналогов
Чат 2: разбить на фазы с [ ] / [x]
Чат 3: критика каждой фазы (лучшее решение? конфликты? безопасность?)
Чат 4+: реализация фаза за фазой
```

Имя файла плана: `plans/YYYY-MM-DD-название-функции.md`

## Workflow: Git + Amvera (правила работы с кодом)

**Стандартный процесс:** Локально → GitHub → Верификация → Деплой по команде

### Фаза 1: Локальная разработка
- Создаём/правим код локально
- **Коммитим часто** (каждый осмысленный шаг)
- Показываю результат владельцу → получаю фидбек
- Переделываем если нужно

### Фаза 2: GitHub
- Пуш в **feature-ветку** (НЕ в main)
- Pull Request
- Обсуждение, правки
- Merge в main после одобрения владельца

### Фаза 3: Верификация (перед деплоем)
- `/code-review` — поиск багов и чистки
- `/verify` — проверка что работает
- `/security-review` — если нужны критичные изменения
- Финальное "Ок" от владельца

### Фаза 4: Деплой на Amvera
- **Только после команды владельца**
- Нажимаем кнопку "Deploy" в Amvera Dashboard
- Ждём пока задеплоится
- Проверяем что сайт живёт

### Критические правила деплоя
1. **Никогда не деплой, пока я не разрешу.**
2. **Всегда коммить — только свои изменения, не трогай чужие.**
3. **Работаем в ветках** — main только для стабильного кода.
4. **Доводи деплой до результата:** отправил в деплой → пока не убедишься, что на сайте актуальная рабочая версия — отслеживай билд, читай логи, находи и чини ошибку, повторяй.

## Безопасность

- **Не читать `.env` и файлы с секретами** (поставить хук-защиту в `.claude/settings.json`)
- Любой устанавливаемый скилл/CLI из интернета — проверять: от создателя ли, нет ли вредоносного кода
- `.gitignore` заранее (`.env`): GitHub помнит всё — утёкшие ключи остаются в истории навсегда
- Не делать внешние запросы без явного разрешения
- Не удалять файлы без подтверждения
- Перед `git push`/деплоем агентов — `/security-review` → `/review`

## Самообучение

В конце каждой сессии проверь:
- узнал ли я что-то новое о пользователе или о работе с ним?
- нарушил ли я какое-то правило, которое теперь надо зафиксировать?

Если да - создай или обнови файл в `ai-clone/feedback/` по формату:
