### Фаза 2: Backend — Модуль выгрузки данных

**✅ Что хорошо:**
- Наследование от `RetailCRMExporter` — переиспользование кода
- Методы названы понятно

**⚠️ Что можно улучшить:**

1. **Конфликт:** Хардкод списка салонов в коде
   - Если салон добавится/переименуется — нужно деплоить
   - **Решение:** Загружать список салонов из конфигурационного JSON файла

2. **Конфликт:** Статусы "Отменен" и "Заказ не доставлен" захардкожены
   - Названия могут измениться в RetailCRM
   - **Решение:** Использовать коды статусов вместо названий, или конфиг файл

3. **Безопасность:** Нет проверки на пустые данные
   - Если API вернёт пустой список — будет ошибка
   - **Решение:** Добавить валидацию данных

**🔄 Предложения:**
- 2.4a: Создать файл `data/salons_config.json` со списком салонов
- 2.4b: Создать файл `data/statuses_config.json` с кодами исключаемых статусов
- 2.13: Добавить метод `_validate_data()` для проверки пустых значений
- 2.14: Добавить логирование для отладки

### Фаза 3: Backend — API эндпоинты

**✅ Что хорошо:**
- RESTful дизайн URL
- Отдельные эндпоинты для разных типов данных

**⚠️ Что можно улучшить:**

1. **Производительность:** Каждый запрос к API = запрос к RetailCRM
   - Если данных много — будет медленно
   - **Решение:** Добавить кэширование в JSON файлы (обновляются раз в час)

2. **Безопасность:** Нет rate limiting
   - Можно положить сервер частыми запросами
   - **Решение:** Добавить кэш (см. выше) или rate limit

3. **UX:** Эндпоинт `/current-month` не принимает параметры
   - Что если нужно посмотреть прошлый месяц?
   - **Решение:** Добавить опциональные параметры `?month=2026-06`

**🔄 Предложения:**
- 3.8: Добавить кэширование в файл `data/sales_cache.json` (ttl = 1 час)
- 3.9: Добавить опциональные параметры для `/current-month`: `?year=2026&month=7`
- 3.10: Добавить endpoint `/api/sales/cache/clear` для принудительного обновления

### Фаза 4: Frontend — HTML дашборд

**✅ Что хорошо:**
- Chart.js — проверенная библиотека
- Бренд-токены уже определены

**⚠️ Что можно улучшить:**

1. **Производительность:** Chart.js из CDN
   - Если CDN недоступен — дашборд не работает
   - **Решение:** Вариантно подключить локальную копию

2. **UX:** Фильтр периода требует два инпута
   - Неудобно выбирать "с 1 по сегодня"
   - **Решение:** Добавить пресеты (Сегоднящий месяц, Прошлый месяц, Этот квартал)

3. **UX:** Нет индикации загрузки
   - Пользователь не знает идёт загрузка или нет
   - **Решение:** Добавить spinner и skeleton loader

4. **Качество:** Форматирование чисел не локализовано
   - "456000" выглядит нечитабельно
   - **Решение:** Использовать `Intl.NumberFormat('ru-RU')`

**🔄 Предложения:**
- 4.3a: Добавить fallback на локальный Chart.js если CDN недоступен
- 4.7a: Добавить пресеты для быстрого выбора периода
- 4.16: Добавить loading spinner
- 4.17: Добавить skeleton для KPI карточек
- 4.18: Использовать `Intl.NumberFormat` для форматирования рублей

## Общая критика архитектуры

### ✅ Сильные стороны
1. **Простота** — HTML + Flask, нет сложных фреймворков
2. **Переиспользование** — используем существующий `RetailCRMExporter`
3. **Бренд** — уже определённые токены

### ⚠️ Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| RetailCRM API недоступен | Средняя | Высокая | Кэширование в JSON |
| CDN с Chart.js недоступен | Низкая | Средняя | Локальная копия |
| Хардкод салонов устареет | Высокая | Низкая | Конфиг файл |
| Медленная загрузка данных | Средняя | Средняя | Кэширование, индикатор |

### 🔧 Рекомендуемые изменения

1. **Добавить кэширование** — критично для производительности
2. **Вынести константы в конфиг** — для гибкости
3. **Добавить loading states** — для UX
4. **Добавить валидацию данных** — для безопасности
