# Процесс: от идеи до запуска

## Путь Миллера (4 шага)

```
1. ИДЕЯ  →  Глубокий анализ + Планирование
2. ПЛАН  →  Разделить на фазы
3. ФАЗЫ  →  Выполнять фазу за фазой (параллельно где возможно)
4. ИТОГ  →  Проверить результат → Запустить → Интегрировать
```

## Шаг 1: Анализ и планирование

Перед тем как писать код — AI делает глубокий анализ задачи.
Миллер использует скилл «пуленепробиваемый» (bulletproof):

```
→ Глубокое исследование задачи и контекста
→ Выявление рисков и неизвестных
→ Составление плана с фазами
→ Уточнение у пользователя перед стартом
```

**Никогда не начинай реализацию без плана.**

## Шаг 2: Разбивка на фазы

Каждая функция = отдельный план-файл в `plans/`.
Каждый план делится на фазы с чекбоксами `[ ]` / `[x]`.

Признак хорошей фазы:
- Можно завершить за один сеанс
- Имеет чёткий проверяемый результат
- Не зависит от незавершённых других фаз (или зависимость явно указана)

## Шаг 3: Выполнение фаз

**Ключевое правило параллельности:**
- Фазы, которые НЕ трогают одни и те же файлы → запускать одновременно (параллельные агенты)
- Фазы, которые работают с одними файлами → только последовательно

Пример:
- Агент A: пишет скрипт сегментации базы клиентов из CRM
- Агент B: параллельно пишет шаблоны сообщений для каждого сегмента
- → Оба готовы → Агент C: интегрирует и тестирует

## Шаг 4: Проверка и запуск

- Запусти и проверь ЧТО получилось, не КАК работает
- Дай обратную связь: «работает» / «не то» / «близко, но нужно изменить X»
- AI итерирует до нужного результата
- После — обязательно обновить план-файл: поставить `[x]` на завершённых фазах

## Режимы работы Claude Code

### Plan Mode (планирование)
Активируется командой `/plan` или через кнопку в интерфейсе.
Claude составляет план и ждёт одобрения — не делает ничего без подтверждения.
Использовать: когда не уверен в подходе, хочешь согласовать перед стартом.

### Action Mode (исполнение)
Обычный режим. Claude сразу делает.
Использовать: когда план понятен и одобрен.

## Одно диалоговое окно = одна задача

Не смешивай задачи в одном чате. Начал работу с базой клиентов — не переключайся на сайт.
Когда контекст наполняется — команда `/compact` сжимает историю.

## Из Потока 4 (уроки 2–3)
*(источники: [[уроки/урок-2-поток-4.md]], [[уроки/урок-3-поток-4.md]])*

### ★★ Итерация «ДОКАЖИ» — против кода ради кода
ИИ зарабатывает на генерации токенов → склонен к «коду ради кода» и раздутым ответам. После любой генерации заставить **ДОКАЗАТЬ**, что это эффективно: «второй проход берёт не то, что сгаллюцинировал, а доказывает — формируется более эффективный ответ». Применять к любому решению, плану, фазе.

### Цикл реализации фичи — расширенный
1. Надиктовка идеи → `idea.md`. 2. Deep Research через оркестратор+субагентов+верификатор. 3. Синтез с «докажи». 4. План в `plans/`. 5. Разбивка на фазы `[ ]`/`[x]`. 6. Валидация фаз (4 вопроса). 7. Доказательство оптимальности плана. 8. Pre-mortem. 9. Промт запуска плана для нового чата. 10. Реализация пофазно: self-check → security review → отметка → следующая. 11. Финал: проверка ошибок + security review + отчёт + локальный запуск.

### 4-вопросная валидация фазы
1. Решает ли проблему? 2. Эффективно? 3. Не код ли ради кода? 4. Не усложнение ли для MVP? → «функция, которая работает уже сегодня, без сверхсложного».

### Pre-mortem — стресс-тест ДО реализации
> «Прошёл год, система провалилась. Думай как будто уже проиграл — найди причины.»

Отсеять нежизнеспособное заранее. «Лучше идею не реализовать, чем целый год в ней ковыряться.»

### ★★ Автономный цикл доведения до результата (деплой до победы)
Агент сам гонит цикл: собрать → билд → деплой → чек → нашёл ошибку в логах → починил → повторил, пока на проде не встанет рабочая версия. «90% случаев чинит сам.» Включается правилом в CLAUDE.md: «пока не задеплоишь в прод — не успокаивайся».

### Самообучающаяся система
Сохранять ВСЁ, что пробовал (что сработало и что нет) в ретроспективы. Перед новой задачей агент СНАЧАЛА читает ретроспективы похожих задач: «это пробовали — нет, а это — да».

### Оценка сроков
Когда ИИ говорит «делать день/неделю/месяц» — он оценивает по человеку. Не верить его временным оценкам.
