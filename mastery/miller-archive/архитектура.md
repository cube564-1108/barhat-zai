# Архитектура проекта по Миллеру

## Стандартная структура папок

```
проект/
├── CLAUDE.md              ← ОБЯЗАТЕЛЕН. Точка входа. Читается AI каждую сессию.
├── .business/             ← Бизнес-контекст (скрытая папка, НЕ в git)
│   ├── wiki/index.md      ← Сжатая карта всего (читать ПЕРВЫМ)
│   ├── company/           ← О компании, команда, юридика
│   ├── products/          ← Продукты, ассортимент, цены
│   ├── audience/          ← Аватар клиента, сегменты, возражения
│   ├── goals/             ← Цели, KPI, прогресс
│   ├── economics/         ← Юнит-экономика, доходы, расходы
│   ├── marketing/         ← Каналы, воронка, конкуренты, контент
│   └── assets/            ← Брендинг (лого, цвета, шрифты)
├── .claude/               ← Настройки и скиллы Claude Code
│   ├── settings.json      ← Настройки (Bypass Permissions, Max Effort)
│   └── skills/            ← Библиотека скиллов (один файл = один скилл)
├── plans/                 ← Технические планы реализации
│   └── YYYY-MM-DD-имя.md  ← Один план = одна функция
└── agents/                ← Определения агентов (опционально)
```

## CLAUDE.md — правила

- Максимум ~200 строк. Больше — контекстное окно наполняется быстрее.
- Содержит: навигацию по проекту, ключевые принципы, где что искать.
- **Не дублируй** в CLAUDE.md то, что уже есть в `.business/`. Только навигация.
- Читается AI автоматически в начале каждой сессии.

## .business/ — правила

- Папка скрытая (точка в начале) = не попадает в git.
- `wiki/index.md` = сжатая карта всего контента. Читается первым, до сырых файлов.
- Сырые файлы в подпапках — только для детализации.
- **Обновляй wiki/index.md** при изменении любого файла в .business/.

## plans/ — правила

- Один план = одна функция/фича/направление
- Имя файла: `YYYY-MM-DD-название-функции.md`
- Структура: фазы → у каждой `[ ]` (не начата) / `[x]` (завершена)
- В конце плана: итоговый блок — реализовано / частично / не начато
- **Обязательное правило:** любой агент ОБЯЗАН обновить план после работы

## .claude/skills/ — правила

- Один файл = один скилл
- Claude читает имя и описание каждого скилла при старте (дёшево по токенам)
- Полное тело скилла загружается только при активации (экономия токенов)
- Не ставь скиллы, которые не нужны — каждый добавляет нагрузку на контекст

## Почему папка скрытая (.business, не business)

- Скрытые папки (начинаются с точки) не попадают в git по умолчанию
- Бизнес-данные, цены, персональные данные клиентов — нельзя в публичный репозиторий
- VS Code показывает скрытые папки если включить «Show Hidden Files»

## Развёртывание: GitHub как связующее звено

```
VS Code (локально)
    ↓ commit
GitHub (облако с историей)
    ↓ push → автотриггер
Amvera / Railway (хостинг)
    ↓
Работающий сайт / бот
```

- GitHub = история всех изменений + точка деплоя
- `commit` = сохранить снимок (локально)
- `push` = отправить в GitHub (в облако)
- После push — хостинг забирает изменения автоматически
- Команда Клоду: «commit и deploy» → он делает всё сам

## Из Потока 4 (уроки 2–3)
*(источники: [[уроки/урок-2-поток-4.md]], [[уроки/урок-3-поток-4.md]])*

### 7 артефактов полного проекта
**(1)** Контекст (структура), **(2)** папка «Бизнес», **(3)** Мастерство (`mastery/`), **(4)** правила CLAUDE.md, **(5)** Скиллы, **(6)** папка «План» (`plans/`), **(7)** Рутина (крон для автономной работы).
- Папка «Бизнес» — только то, что РАБОТАЕТ; в каждой подпапке `INDEX.md`.
- AI-клон — **один на все проекты**, наполнять при реальном фундаменте личности.

### ★ Три критические строчки CLAUDE.md (контроль деплоя)
1. «Никогда не деплой, пока я не разрешу.»
2. «Всегда коммить — только свои изменения, не трогай чужие.»
3. «При деплое — пока не убедишься, что на сайте актуальная версия: отслеживай, находи ошибку, лечи, пока не задеплоишь.»

### GitHub — детально
- **Репозиторий** = папка проекта в облаке (приватный = ты + команда; публичный = любой скачает).
- **Commit** = сохранение с описанием «кто/когда/что/зачем». **Push** = отправка в облако.
- Главная ценность — **история для отката**.

### Commit / Reset / откат
Коммитить ЧАСТО (иначе параллельные агенты перепишут код друг друга — откатить нечего). Откат: «найди коммит, где появилось изменение» → diff (было/стало) → reset → коммит отката. Раздел **Source Control** в VS Code показывает коммиты и что задеплоено/нет.

### `.gitignore` — «GitHub помнит всё»
Закладывать ЗАРАНЕЕ, до первого пуша: `.env`/ключи, `.business/`, локальные конфиги. Утёкшие данные остаются в истории навсегда. «Поимели ключи — поимели проект.»

### Методология дизайна в CLAUDE.md
Файл «методология дизайна» (фирменные цвета, стиль, правила) + ссылка в CLAUDE.md → каждая новая страница в едином стиле без повторения референсов.
