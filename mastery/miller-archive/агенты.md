# Мультиагентные системы

## Принцип

Один оркестратор координирует несколько специализированных агентов.
Агенты, работающие с разными файлами, запускаются параллельно — это даёт ×5 к скорости.

```
Оркестратор
├── Агент A (исследование)     ─┐
├── Агент B (валидация)         ├── параллельно
└── Агент C (синтез)           ─┘
         ↓
    Оркестратор собирает результат
         ↓
    Финальный ответ / следующий шаг
```

## Паттерн «пуленепробиваемый» (bulletproof) от Миллера

Скилл, который Миллер использует для любой серьёзной задачи:

```
1. Глубокое исследование задачи и контекста
2. Составление детального плана с фазами
3. Запуск параллельных агентов для независимых фаз
4. Верификация каждого результата
5. Финальная интеграция и деплой
```

## Типы агентов

### Оркестратор
- Читает утренний дайджест / входящие данные
- Решает что передать каким агентам
- Собирает итоговый отчёт для человека
- Эскалирует только то, что требует решения человека

### Исследователь (Researcher)
- Собирает данные из источников
- Не принимает решений — только добывает факты
- Пример: тянет данные из CRM, веб-аналитики, рекламных кабинетов

### Валидатор (Validator)
- Проверяет результат другого агента
- Ищет ошибки, несоответствия, исключения
- Пример: проверяет корректность сегментации клиентов

### Синтезатор (Synthesizer)
- Берёт данные от нескольких агентов
- Формирует единый вывод / отчёт / решение

### Исполнитель (Executor)
- Выполняет конкретное действие
- Пример: отправляет сообщения, создаёт записи в CRM, публикует контент

## Правила параллельного запуска

**Можно параллельно:**
- Агенты, которые читают из разных источников
- Агенты, которые пишут в разные файлы
- Независимые задачи без shared state

**Только последовательно:**
- Агент B зависит от результата Агента A
- Оба агента пишут в один и тот же файл
- Второй агент должен проверить работу первого

## Пример: запуск еженедельного отчёта

```
Оркестратор запускает параллельно:
├── Агент АНАЛИТИКА → тянет данные CRM за неделю
├── Агент РЕКЛАМА → тянет данные рекламных кабинетов за неделю
└── Агент КОНТЕНТ → тянет охваты соцсетей за неделю

Все три готовы →
Агент СИНТЕЗАТОР → объединяет в единый дайджест → отправляет владельцу
```

## Практика: как запускать агентов в Claude Code

1. Описать оркестратору общую задачу
2. Оркестратор сам разобьёт на подзадачи и запустит агентов
3. Или явно написать: «запусти параллельно агента X и агента Y»
4. Результат каждого агента пишется в файл → оркестратор читает все файлы → собирает итог

## Память агентов

Агенты не помнят предыдущие сессии автоматически.
Поэтому вся важная информация хранится в файлах:
- `.business/wiki/index.md` — общий мозг
- `plans/` — статус задач
- Агент читает файлы в начале работы → пишет результаты в конце работы

### Уточнённая иерархия агентов

| Уровень | Роль |
|---|---|
| **Агент** | ИИ с инструкцией под задачу |
| **Субагент** | порождается оркестратором, делает узкую задачу, возвращает результат и закрывается |
| **Оркестратор** | дирижирует субагентами: сам пишет им задачи, запускает, следит, собирает |
| **Мультиагент** | промежуточная «ячейка»: у него своё дело, и он ПОД СОБОЙ ещё запускает субагентов |

Пример: SEO-агент = 7 мультиагентов × свои субагенты → ~25 субагентов параллельно.

### Главный сдвиг: автономные агенты, а не «стартапчики»
Делать то, что работает каждый день само. Автономный агент = многоагентная система (оркестратор + мультиагенты + субагенты), работающая сутками ради цели. Каждую задачу выстраивать «вокруг автономности»: один раз сделал → работает вечно.

### ★ OpenRouter vs обвязка вокруг Claude Code
- **OpenRouter / API-ключи** — только для **простых** задач (один-два вызова).
- Для **сложных автономных агентов** (60 мин — 10 часов, оркестрация субагентов, 14+ итераций) — **обвязка вокруг подписки Claude Code / Qwen, локально или на VPS**, не через API-ключ («и денег не напасёшься, и многоитеративность не сработает»).

### Промт оркестрации многофазного плана (шаблон)
> «1) прочитай план; 2) ты оркестратор — запускай субагентов на каждую фазу; 3) субагент закончил → сам проверяет себя на ошибки + security review → сообщает; 4) оркестратор проверяет, делает security review, ставит отметку, запускает следующего; 5) после ВСЕХ фаз — финальный агент проверки ошибок + финальный security review + детальный отчёт со ссылкой на локальный запуск.»
