**Кто:** владелец сети клиник лазерной эпиляции, автор курса «Смыслокодинг».
Построил больше автоматизации за 6 месяцев, чем за 5 предыдущих лет.
Не программист — описывает AI словами, AI пишет код.

**Подробный архив:** [[mastery/miller-archive/index.md]]

## Метод 1: iCLON — цифровая проекция владельца

**Суть:** папка `ai-clone/` в корне проекта. Агент читает её перед любой задачей
и действует от имени владельца — с его ценностями, стилем, логикой решений.

**Алгоритм создания:**
1. Создать структуру `ai-clone/` (identity/, voice/, thinking/, principles/, feedback/, style/)
2. Провести голосовое интервью по одному блоку за раз: «Ты — интервьюер. Собираешь мой профиль для AI-Clone»
3. После каждого блока сразу записать в соответствующий файл
4. В `feedback/` — не заполнять руками, только из реальных ошибок: «запиши правилом в формате Правило / Почему / Как применить»

**Когда применять:** при любой задаче, где важен голос, стиль или решение от имени владельца. iCLON — один на все проекты.

**Подробнее:** [[mastery/miller-archive/уроки/урок-2-поток-3.md]] → раздел «iCLON»

## Метод 2: Контекст инжиниринг

**Суть:** агент ходит по вики-ссылкам как по паутине — берёт только нужные файлы
под конкретную задачу, не загружает всё подряд. Один короткий промт → огромная работа.

**Алгоритм:**
1. Каждая папка имеет `INDEX.md` — карту содержимого
2. Файлы ссылаются друг на друга через `[[имя-файла]]`
3. Агент читает INDEX.md первым, потом проваливается только в нужные файлы
4. Никакого хардкода данных — живые цифры через `life-metrics.md` с маркером **LIVE**
5. Каждый файл не более 200 строк — если больше, дробить

**Когда применять:** при проектировании любой новой папки или документа в проекте.

**Подробнее:** [[mastery/miller-archive/уроки/урок-2-поток-3.md]] → раздел «Контекст инжиниринг»
[[mastery/miller-archive/контекст.md]] → «вторая память, ОБ которую думаешь»

## Метод 3: Multi-chat workflow

**Суть:** один чат = одна задача. Планирование, фазы и реализация — в разных чатах.
Иначе история диалога раздувается и токены улетают впустую.

**Алгоритм:**
1. **Чат 1** — создать черновой план, попросить поискать как другие решали
2. **Чат 2** — разбить план на фазы с чекбоксами `[ ]`/`[x]`
3. **Чат 3** — критика каждой фазы: лучшее решение? конфликты? безопасность?
4. **Чат 4+** — реализация фаза за фазой (параллельные агенты где можно)
5. Только Claude Code (CLI) умеет писать файлы — финальная запись всегда здесь

**Когда применять:** при любой задаче крупнее 30 минут работы.

**Подробнее:** [[mastery/miller-archive/уроки/урок-2-поток-3.md]] → раздел «Правило одна задача — один диалог»
[[plans/README.md]] → правила оформления планов

## Метод 4: Bulletproof-цикл для крупных фич

**Суть:** перед реализацией любой крупной функции — прогон через скилл Bulletproof.
Проверяет план по 5 критериям до начала работы.

**Алгоритм:**
1. Установить скилл Bulletproof глобально в `~/.claude/skills/`
2. При старте крупной фичи вызвать: `/bulletproof`
3. Скилл проверит: интернет-ресёрч → техническая реализуемость → конфликты с другими фазами → безопасность → соответствие бизнес-целям
4. Принять правки → только потом реализация

**Когда применять:** любая задача, которую нельзя откатить за 5 минут.

**Подробнее:** [[mastery/miller-archive/скиллы.md]]

## Метод 5: Мастерство — личный совет директоров

**Суть:** папка `mastery/` с методологиями авторов. Агент применяет конкретные
приёмы, а не общие шаблоны. Один файл = один автор, 1-3 метода с алгоритмом.
Скилл = «сделай по шагам»; мастерство = «применяй проверенные подходы по триггер-слову».

**Алгоритм наполнения:**
1. Выбрать одну область (где самая частая задача сейчас)
2. Найти 3-5 авторов, которых реально читают практики — **делателей с цифрами, не говорунов**
3. По каждому: кто + 1-3 метода с шагами «что→как→когда» (для практиков — скормить их реальный контент: «выдели повторяющиеся паттерны»)
4. Создать `mastery/<область>/<автор>.md` + `INDEX.md` области, задать триггер-слово
5. Ссылаться на подробные источники через `[[вики-ссылку]]`

**Когда применять:** при задачах, где нужна методология (контент, маркетинг, продажи, операции).

**Подробнее:** [[mastery/miller-archive/уроки/урок-2-поток-3.md]] → раздел «Папка Мастерство»
[[mastery/miller-archive/скиллы.md]] → «Скилл vs Мастерство»

## Метод 6: Оркестратор реализации

**Суть:** два чата — один оркестратор (знает план, пишет промты), один исполнитель (реализует одну фазу). Экономит токены, исключает «кашу» кода при параллельных агентах.

**Алгоритм:**
1. Чат-оркестратор: «Ты оркестратор. Прочитай план. Пиши мне промт для каждой фазы по одному.»
2. Чат-исполнитель: получает промт, реализует фазу, отчитывается
3. Оркестратор проверяет результат → пишет промт следующей фазы
4. Цикл до конца. Каждая фаза → коммит в GitHub.
5. Если фазы независимы → несколько исполнителей параллельно (только после явного ОКА от оркестратора)

**Иерархия (Поток 4):** агент → субагент (узкая задача, закрылся) → оркестратор (дирижирует) → мультиагент (запускает субагентов под собой).

**Когда применять:** любая задача с планом из 3+ фаз. Для больших проектов — обязательно.

**Антипаттерн:** «сделай всё сам» → агент запускает неконтролируемых субагентов → они переписывают код друг друга → проект разваливается.

**Подробнее:** [[mastery/miller-archive/уроки/урок-3-поток-3.md]] → раздел «Оркестратор реализации»
[[mastery/miller-archive/агенты.md]] → «OpenRouter vs обвязка вокруг Claude Code»

## Метод 7: Bulletproof при создании скиллов — обязательный шаг

**Суть:** скилл — это архитектурное решение. Неверно сформулированный скилл используется неправильно везде, где вызывается. Поэтому создание любого нового скилла проходит через `/bulletproof` до записи файла.

**Алгоритм:**
1. Получена задача «создать скилл для X» → сначала `/bulletproof`
2. Bulletproof проверяет: нет ли уже похожего скилла? задача повторяется ≥ 3 раз/мес? скилл решает одну задачу, не три?
3. Только после зелёного света → `/skill-creator`
4. После создания скилла — `/bulletproof` снова на первый реальный вызов (тест-прогон)

**Базовый набор скиллов:** `/bulletproof` (перед крупной задачей и созданием скилла), `/skill-creator` (формализовать повторяющуюся задачу).

**Когда применять:** при создании любого нового скилла — без исключений.

**Подробнее:** [[.claude/skills/bulletproof.md]] · [[.claude/skills/skill-creator.md]]

## Метод 8: IDEA.md + Deep Research перед планом

**Суть:** перед созданием плана для крупной фичи — сначала параллельный ресёрч, синтез в `IDEA.md`, итерации критики. Иначе план строится на интуиции, а не на мировом опыте.

**Алгоритм:**
1. Запустить Deep Research через оркестратор + субагентов (Perplexity / Google / ChatGPT, или один оркестратор с субагентами): «кто решал эту проблему, как, что сработало, а что нет»
2. Финальный субагент проверяет находки на ключевую цель и оставляет только то, что реально пригодится — должен **доказать**, что это не код ради кода
3. Синтезировать в `IDEA.md` (рядом с планом или в `plans/`)
4. Итерации валидации: отсечение лишнего → критика каждой идеи → «1 строка суть + 1 строка зачем»
5. Только после утверждённого `IDEA.md` → создавать план с фазами

**Когда применять:** любая задача длиннее одной сессии. Для однодневных правок — сразу план.

**Антипаттерн:** «придумали — сделали». Без ресёрча — изобретаем велосипед.

**Подробнее:** [[mastery/miller-archive/уроки/урок-3-поток-3.md]] → раздел «Deep Research»
[[mastery/miller-archive/уроки/урок-2-поток-4.md]] → промпт Deep Research

## Метод 9: Итерация «докажи» + валидация плана

**Суть:** ИИ зарабатывает на генерации токенов → склонен к «коду ради кода» и раздутым ответам. Лечится принуждением ДОКАЗАТЬ эффективность на каждом шаге плана.

**Алгоритм (применять к любому плану до реализации):**
1. **4 вопроса к каждой фазе:** решает ли проблему? эффективно? не код ли ради кода? не усложнение ли для MVP?
2. **«Докажи»:** «Я так думаю — а ты докажи, что это достойно отбора / что это не борщ, а ресторанное блюдо.»
3. **Оптимальность:** «Докажи, что этот план — лучшее решение. Продумали ли на шаги вперёд?»
4. **Pre-mortem:** «Прошёл год, система провалилась. Думай как будто уже проиграл — найди причины.» → отсеять нежизнеспособное ДО недель работы.

**Когда применять:** перед реализацией любого плана и при отборе решений из Research. Встроить в шаблон `plans/`.

**Подробнее:** [[mastery/miller-archive/уроки/урок-2-поток-4.md]] → «Докажи», «Pre-mortem»

## Метод 10: Автономный цикл доведения до результата (деплой до победы)

**Суть:** с думающими моделями деплой не итерируется руками — агент сам гоняет цикл «билд → деплой → чек → почини ошибку → повтори», пока на проде не встанет рабочая версия. «90% случаев чинит сам.»

**Алгоритм:**
1. Прописать в `CLAUDE.md` три правила: (1) не деплоить без разрешения; (2) коммитить только свои изменения; (3) «пока не задеплоишь в прод — не успокаивайся: отслеживай билд, читай логи, чини, повторяй».
2. Коммитить ЧАСТО (иначе параллельные агенты перепишут код друг друга — откатить нечего).
3. При проблеме — откат через историю коммитов (найти коммит → diff → reset).
4. `.gitignore` закладывать ЗАРАНЕЕ (GitHub помнит всё; утёкшие ключи = угнанный проект).

**Когда применять:** любой деплой.

**Подробнее:** [[mastery/miller-archive/уроки/урок-3-поток-4.md]] → «Автономный цикл»
[[mastery/miller-archive/архитектура.md]] → «Три критические строчки CLAUDE.md»

## Метод 11: Самообучающаяся система + «думать об вторую память»

**Суть:** база знаний нужна не только чтобы ИИ читал контекст, но чтобы ОБ неё думать. Плюс — сохранять всё, что пробовал, иначе повторяешь старые ошибки.

**Алгоритм:**
1. **Ретроспективы:** после каждой задачи сохранять что сработало и что нет (`.business/история/`). Перед новой задачей агент СНАЧАЛА читает релевантные ретроспективы: «это пробовали — нет, а это — да».
2. **Ежемесячный агент x2:** «Что ключевое сделать в этом месяце, чтобы x2 от прошлого? Проанализируй 80/20: что дало 80% результата (оставить), что неэффективно (убрать). Читай папку Бизнес, будь оркестратором, думай в несколько итераций.»
3. Цифры держать в контексте, в файлы хардкодом не писать (живые данные — через API/LIVE).

**Когда применять:** ежемесячно (агент x2) + перед каждой крупной фичей (чтение ретроспектив).

**Подробнее:** [[mastery/miller-archive/уроки/урок-2-поток-4.md]] → «Самообучающаяся система», «Думать об вторую память»
