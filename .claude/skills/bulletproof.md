## Цель

Поймать критические ошибки в плане **до** того как начнётся код. Один прогон через Bulletproof экономит 2–10 часов переработки.

---

## Когда запускать

- Любая задача крупнее 30 минут работы
- Новый агент, триггер, рассылка
- Изменение архитектуры или схемы данных
- Деплой на хостинг / пуш в main
- Создание нового скилла (обязательно)

---

## Алгоритм — 5 критериев

Прочитай план (файл из `plans/` или описание задачи в чате). Пройди каждый критерий последовательно.

### Критерий 1 — Ресёрч: как другие это решали?

1. Найди 2–3 примера как аналогичную задачу решали (Perplexity, WebSearch, GitHub)
2. Запиши: что сработало, что не сработало, какие подводные камни
3. Вопрос: **есть ли готовое решение лучше чем то что планируем?**
4. Если да — предложи его вместо плана. Если нет — продолжай.

### Критерий 2 — Реализуемость: технически можно?

1. Проверь каждый технический компонент плана:
   - API существует и доступен с выбранного хостинга?
   - Зависимости (пакеты, библиотеки) поддерживаются на целевой платформе?
   - Есть ли credentials / токены / доступы для этого?
2. Оцени сложность: часы / дни / недели
3. Вопрос: **что первым сломается при реализации?**
4. Если риск высокий — предложи упрощённую версию для первой итерации.

### Критерий 3 — Конфликты: что сломает в проекте?

1. Прочитай `plans/` — есть ли пересечения с другими активными планами?
2. Проверь: не конфликтует ли с уже работающими агентами и триггерами?
3. Проверь зависимости данных: если изменяется схема — что перестанет работать?
4. Вопрос: **какие три вещи могут сломаться из-за этого изменения?**
5. Для каждой — план отката или защита.

### Критерий 4 — Безопасность: нет ли дыр?

1. Проверь по чеклисту:
   - [ ] Нет секретов / токенов в коде (только в .env)
   - [ ] SQL-запросы параметризованы (нет конкатенации строк)
   - [ ] API-ключи не логируются
   - [ ] Входящие данные (от пользователя / webhook) валидируются
   - [ ] Нет open redirect, XSS, command injection
   - [ ] Персональные данные клиентов не уходят в сторонние сервисы без согласия
2. Если найдена уязвимость — стоп. Исправить до реализации.

### Критерий 5 — Бизнес-цели: зачем это проекту?

1. Прочитай `.business/wiki/index.md` — текущие приоритеты
2. Ответь на три вопроса:
   - Какую метрику из `goals/` эта задача двигает?
   - Насколько это автоматизирует владельца (0–10)? Если < 5 — пересмотри.
   - Это приоритетное направление проекта или нет?
3. Вопрос: **если не сделать это сейчас — что потеряем?**
4. Если ответ «ничего критичного» — задача кандидат на перенос или удаление.

## Вывод

После прохождения всех критериев выдай одно из двух:

### Зелёный свет ✅
```
Bulletproof пройден. Критических проблем нет.
Найденные замечания: [список если есть]
Можно реализовывать.
```

### Стоп 🔴
```
Bulletproof: СТОП.
Критерий [N]: [описание проблемы]
Необходимые правки перед реализацией:
1. ...
2. ...
Вернись к /bulletproof после правок.
```
